# Figura 与 Sparkle-Morpher 同步机制对比

本文基于当前仓库源码分析 Figura 主项目、后端 `sculptor`，以及 `Sparkle-Morpher` 的同步设计。结论先行：**Figura 不得照搬 Sparkle-Morpher 的同步机制**。两者目标、网络边界、服务端假设和运行时模型都不同；直接移植 Sparkle-Morpher 的实时游戏服同步，会破坏 Figura “无服务器 Mod 也能让其他玩家看到 Avatar”的核心初衷。

## 一、项目整体定位

### Figura

Figura 是客户端 Avatar 平台。README 明确强调：玩家可以高度自定义模型，并且其他玩家能看到 Avatar，**不要求 Minecraft 服务器安装 Mod**。

当前 Figura 的同步核心由两部分组成：

- 客户端：`common/src/main/java/org/figuramc/figura/backend2/NetworkStuff.java`
- 后端：`sculptor/src/api/figura/*`

它的同步重点不是把 Minecraft 实体状态逐帧复制给旁观客户端，而是：

1. 上传和下载 Avatar 资产；
2. 查询玩家装备的 Avatar 与徽章信息；
3. 通过 WebSocket 订阅目标玩家；
4. 在 Avatar 发生变更时推送 reload 事件；
5. Lua `pings` 只同步脚本主动发出的小型事件数据。

### sculptor

`sculptor` 是 Figura 的中心后端，负责 HTTP API、WebSocket 会话、认证、Avatar 文件存储、订阅关系和广播事件。

关键文件：

- `sculptor/src/main.rs`
- `sculptor/src/api/figura/profile.rs`
- `sculptor/src/api/figura/websocket/handler.rs`
- `sculptor/src/api/figura/websocket/types/c2s.rs`
- `sculptor/src/api/figura/websocket/types/s2c.rs`
- `sculptor/src/state/state.rs`

它不是 Minecraft 游戏服 Mod，也不持有游戏世界实体状态。它只知道用户、token、Avatar 文件、session 和订阅者。

### Sparkle-Morpher

Sparkle-Morpher 是 Minecraft 自定义模型加载器，README 中说明它有服务端功能，可以让服务器定义模型清单、向客户端推送模型，并通过实体数据进行客户端-服务端模型状态同步。

关键文件：

- `Sparkle-Morpher/common/src/main/java/com/micaftic/morpher/network/NetworkHandler.java`
- `Sparkle-Morpher/common/src/main/java/com/micaftic/morpher/network/sync/PlayerStateSynchronizer.java`
- `Sparkle-Morpher/common/src/main/java/com/micaftic/morpher/network/message/S2CSyncPlayerStatePacket.java`
- `Sparkle-Morpher/common/src/main/java/com/micaftic/morpher/network/message/S2CSetModelAndTexturePacket.java`
- `Sparkle-Morpher/common/src/main/java/com/micaftic/morpher/client/animation/molang/functions/ysm/Sync.java`

它的同步重点是 Minecraft 游戏服内的实时实体状态、模型选择、Molang 变量、动画表达式和模型上传分片。

## 二、Figura 同步机制

### 1. Avatar 资产同步

Figura 客户端通过 HTTP 与后端交互。`HttpAPI` 定义了这些核心接口：

- `GET /api/:uuid`：查询用户资料、装备 Avatar、徽章、版本等；
- `GET /api/:uuid/avatar`：下载 Avatar；
- `PUT /api/avatar`：上传当前用户 Avatar；
- `DELETE /api/avatar`：删除当前用户 Avatar；
- `POST /api/equip`：装备 Avatar 并触发事件；
- `GET /api/limits`、`/version`、`/motd`：后端限制、版本、公告。

客户端 `UserData.loadData` 会先用 hash 检查本地缓存，命中则直接加载；未命中才调用 `NetworkStuff.getAvatar` 下载。也就是说，Figura 资产同步是**按需拉取 + hash 缓存**，不是持续推送完整模型。

### 2. WebSocket 会话与订阅

Figura WebSocket 协议非常小：

C2S：

- `Token`：认证；
- `Ping`：Lua ping；
- `Sub(UUID)`：订阅某个玩家；
- `Unsub(UUID)`：取消订阅。

S2C：

- `Auth`：认证成功；
- `Ping(UUID, id, sync, data)`：转发 Lua ping；
- `Event(UUID)`：某玩家 Avatar 更新；
- `Toast`、`Chat`、`Notice`：提示和通知。

客户端 `FiguraWebSocketAdapter` 连接后发送 token，收到 `Auth` 后调用 `NetworkStuff.subscribeAll()`；收到 `Event(UUID)` 后执行 `AvatarManager.reloadAvatar(uuid)`，也就是清理缓存并重新走 HTTP 查询/下载。

`sculptor` 端 `handler.rs` 对订阅的处理是：每个被订阅用户有一个 broadcast channel，订阅者通过 `Sub(UUID)` 监听该 UUID 的广播；上传、装备、删除 Avatar 时，`profile.rs` 的 `send_event` 向订阅者和用户本人发送 `S2CMessage::Event(uuid)`。

### 3. Lua pings

Figura 的 Lua pings 是 Avatar 脚本层的显式小消息。README FAQ 也说明：pings 从宿主玩家发送，用于同步按键、动作轮等事件，内容和大小受限制并有速率限制。

客户端侧 `S2CMessageHandler.ping` 收到 ping 后找到对应 Avatar，执行 `avatar.runPing(id, data)`。后端只做转发和 echo，不解释数据内容。

因此，Figura 的同步边界很明确：

- 后端不理解 Minecraft 实体；
- 后端不计算动画；
- 后端不保存运行时骨骼/姿态/输入状态；
- 实时脚本同步只通过受限 ping 完成。

## 三、Sparkle-Morpher 同步机制

### 1. 游戏服自定义网络通道

`NetworkHandler` 注册了大量自定义 packet，包含：

- 模型同步 payload；
- 模型切换；
- 授权模型和收藏模型同步；
- projectile、vehicle 模型同步；
- Molang 执行；
- 动画表达式同步；
- 玩家状态同步；
- 模型上传 start/chunk/finish/result；
- 版本握手。

通道版本为 `2.6.0`，连接有效性依赖客户端和服务端都能完成该通道握手。换言之，Sparkle-Morpher 假设 Minecraft 服务器侧有对应 Mod/网络通道能力。

### 2. 玩家状态增量同步

`PlayerStateSynchronizer` 在服务端跟踪玩家状态变化，并把变化合并进 `S2CSyncPlayerStatePacket`。同步字段包括：

- flying；
- 经验等级；
- 血量和最大血量；
- 饥饿值；
- strafe / vertical / forward 输入；
- 盾牌格挡冷却；
- 药水效果；
- 当前模型切换 id；
- Molang 变量。

`S2CSyncPlayerStatePacket` 用 bit flags 标记哪些字段存在，只编码变化字段；full sync 时携带完整状态。发送路径使用 `NetworkHandler.sendToTrackingEntityAndSelf`，即发给追踪该实体的玩家和自己。

这是典型的 Minecraft 实体状态同步模式：服务端知道实体，服务端决定向哪些客户端广播，客户端按 entityId 应用状态。

### 3. 模型和动画同步

`S2CSetModelAndTexturePacket` 直接给 entityId 设置模型和贴图，同时附带一次玩家状态同步。

`Sync.java` 的 `ysm.sync(...)` 会在本地玩家执行时发送 `C2SSyncAnimationExpressionPacket` 到服务端，再由服务端转成 `S2CSyncAnimationExpressionPacket` 广播给追踪者，客户端调用 `PlayerCapability.executeAnimationExpression(...)`。

这是一套游戏服中继的实时动画表达式同步。

### 4. 分片上传

Sparkle-Morpher 还有 `C2SModelUploadStartPacket`、`C2SModelUploadChunkPacket`、`C2SModelUploadFinishPacket` 和 `S2CModelUploadResultPacket`，由 Minecraft 服务器接收模型上传并管理结果。README 也提到服务器带宽限制配置，说明模型传输可通过服务器网络通道进行限速。

这与 Figura 的 HTTP 上传完全不同。

## 四、核心差异

| 维度 | Figura + sculptor | Sparkle-Morpher |
| --- | --- | --- |
| 服务端类型 | 独立中心后端 | Minecraft 游戏服务器 Mod 网络通道 |
| 是否要求服务器装 Mod | 不要求 | 实时同步能力依赖服务器通道 |
| 同步对象 | Avatar 文件、装备状态、徽章、Lua ping、reload 事件 | 实体状态、模型选择、Molang 变量、动画表达式、模型分片 |
| 数据拉取方式 | HTTP 按需下载，hash 缓存 | packet 推送和实体追踪广播 |
| 事件通知 | WebSocket `Event(UUID)` 后重新查询/下载 | 直接向追踪实体的客户端发 S2C packet |
| 实时状态 | 只允许脚本显式 ping 小数据 | 服务端主动同步玩家状态和模型状态 |
| 后端是否理解 Minecraft 世界 | 不理解 | Minecraft 服务器天然理解实体和追踪关系 |
| 设计目标 | 跨服务器、无服务器 Mod 依赖、Avatar 平台 | 服务器内模型加载、管理和实时状态一致 |

## 五、为什么 Figura 不得照搬 Sparkle-Morpher 的同步机制

### 1. 会破坏 Figura 的核心初衷

Figura 的关键价值是：玩家不需要所在 Minecraft 服务器安装 Figura，也能让其他安装 Figura 的客户端看到 Avatar。它通过中心后端和客户端订阅机制绕开了游戏服 Mod 依赖。

Sparkle-Morpher 的同步机制则建立在 Minecraft 服务器可注册自定义网络包、可识别实体追踪关系、可广播 S2C packet 的前提上。照搬后，Figura 会从“客户端 Avatar 网络”退化为“服务器 Mod 驱动的模型同步”，这与 Figura 初衷冲突。

### 2. 后端语义完全不匹配

`sculptor` 只保存用户、token、Avatar 文件和 WebSocket session。它没有 `ServerPlayer`、entityId、tracking players、世界 tick，也不能知道某个玩家的飞行、血量、输入、药水效果。

Sparkle-Morpher 的 `PlayerStateSynchronizer` 依赖 `ServerPlayer` 和 Minecraft 服务端实体状态。把这套机制搬到 `sculptor` 需要重新发明一个跨服务器实体状态系统，既不准确，也不可靠。

### 3. 会显著增加带宽和隐私风险

Figura 当前只同步 Avatar 资产变更事件和脚本主动发出的 ping。Sparkle-Morpher 会同步更多运行时状态，例如输入、血量、饥饿值、效果、Molang 变量等。

对 Figura 来说，这些状态并不是全局 Avatar 分发所必需的。强行引入会扩大数据面、增加中心后端压力，也会让玩家状态暴露给不应接收这些信息的上下文。

### 4. 会削弱现有缓存模型

Figura 的 Avatar 资产通过 hash 缓存和按需下载减少重复传输。Sparkle-Morpher 的服务器分片上传和 S2C 模型同步更适合服务器管理模型资源，而不是中心后端长期分发 Avatar 文件。

Figura 应继续保留“资产走 HTTP/缓存，变更走 WebSocket 事件，行为走 ping”的三层结构。

### 5. 与 Lua ping 的权限和限流模型冲突

Figura 的 Lua ping 是 Avatar 作者显式调用、受权限和速率限制的小消息。Sparkle-Morpher 的动画表达式同步更偏向 YSM/Molang 运行时的即时表达式广播。

如果 Figura 直接引入 `ysm.sync` 式的高频表达式同步，容易绕过现有 ping 语义，扩大脚本同步能力，增加滥用面，并破坏 Figura 对脚本行为的可控边界。

## 六、推荐的 Figura 同步原则

Figura 后续若增强同步，应遵循以下原则：

1. **保留中心后端模型**：继续让 `sculptor` 负责 Avatar 文件、用户资料、认证、订阅和事件通知。
2. **不依赖 Minecraft 服务器安装 Mod**：任何同步增强都应能在普通服务器上工作。
3. **资产与运行时分离**：Avatar 文件仍通过 HTTP/hash/cache；运行时小事件仍通过 WebSocket ping。
4. **只同步 Avatar 需要的最小数据**：不要默认同步玩家输入、血量、状态效果等 Minecraft 运行时状态。
5. **事件优先，状态慎用**：Avatar 变更使用 `Event(UUID)` 触发 reload；脚本行为使用显式 ping；不要引入持续实体状态镜像。
6. **权限、限流、大小限制必须保留**：任何新同步消息都必须受权限和速率控制。
7. **可兼容 YSM，但不可被 YSM 反向改造架构**：YSM/Molang 适配应嵌入 Figura 的 Avatar 生命周期，而不是让 Figura 变成 Sparkle-Morpher 式服务器同步系统。

## 七、可借鉴但不能照搬的点

Sparkle-Morpher 仍有一些工程经验可参考：

- bit flags 增量编码可用于未来 Figura 小型结构化 ping，但必须受 Figura 权限和限流约束；
- 分片传输的进度、校验、结果回执思想可用于改善 HTTP 上传体验，但不应改成 Minecraft packet 上传；
- full sync / delta sync 的概念可用于客户端本地 runtime 状态恢复，但不应变成中心后端的实体状态镜像；
- 版本握手思想可用于 WebSocket 协议升级，但不能要求游戏服务器参与。

结论：**Figura 可以吸收 Sparkle-Morpher 的局部编码和可靠性思想，但同步架构必须坚持 Figura 自己的模型：中心后端 + HTTP 资产 + WebSocket 订阅事件 + 受限 Lua ping。不得照搬 Sparkle-Morpher 的 Minecraft 服务器实时同步机制。**
