# Figura YSM 与 Sparkle-Morpher 动画机制对齐分析

本文只比较“动画机制相关”实现：动画资源加载、基础/并行动画、动画控制器、轮盘/额外动作触发、Molang 绑定、timeline/event、显隐与 scale 混合、projectile/vehicle 子模型动画。模型解析、普通渲染、UI 美术和网络同步只在影响动画触发时提及。

## 结论概览

| 模块 | 当前状态 | 结论 |
| --- | --- | --- |
| 新版/旧版 YSM 动画资源加载 | 基本一致 | Figura 已能加载 `main/arm/extra/slashblade/tac` 等动画文件，也补了旧版 `main.json` 的 `extra_animation_names`。 |
| `extra_animation` 轮盘动作生成 | 基本一致 | Figura 已解析 `properties.extra_animation`、`extra_animation_classify`、`extra_animation_buttons`，旧版动作也能进轮盘。 |
| 基础隐藏与 extra 动作显示 | 基本一致，仍需实机验证 | 已按 Sparkle 语义修正：`pre_parallel/parallel` 的零 scale 可作为默认隐藏，extra 动作非零 scale 可重新显示。 |
| timeline `v.xxx=...` 执行 | 部分一致 | Figura 已支持 timeline 表达式执行，但事件时序、剩余事件补发、loop 边界仍弱于 Sparkle 的 executor。 |
| animation controller 解析与运行 | 部分一致 | Figura 有 JSON controller state/transition/animations/on_entry/on_exit 支持，但控制器槽位体系、transition、loop mode 和 controller context 仍不完整。 |
| native player 状态动画 | 部分一致 | Figura 有 idle/walk/run/sneak/jump/fall/swim/ride/sleep 等基础状态，但 Sparkle 的 controller/predicate 覆盖面更广。 |
| Molang/ctrl/ysm 查询变量 | 部分一致 | Figura 已补一批关键变量和 discovered controls，但 Sparkle 的 `YSMBinding`/`CtrlBinding` 覆盖大量装备、武器、船、光照、mod compat 查询。 |
| sound/particle event | 部分一致 | Figura 已支持 `sound`、`particle` 基础事件，但 Sparkle 有专门 keyframe executor 和资源包 sound/effect 管线。 |
| projectile 子模型动画 | 部分一致 | Figura 已加载 `files.projectiles` 并接入 arrow/trident 静态渲染入口，但 projectile 独立动画 controller 尚未完全对齐。 |
| vehicle 子模型动画 | 未完全对齐 | Figura 已加载 `files.vehicles` 资源，但没有完整 vehicle entity hook、origin/ride/move controller。 |
| Geckolib 过渡/混合队列 | 未完全对齐 | Sparkle 使用 `AnimationControllerInstance`、bone queue、begin/end transition、scale special transition；Figura 目前是简化加权叠加。 |
| 网络同步动作触发 | 当前阶段不对齐 Sparkle 协议 | 不能要求 Minecraft 服务器安装 YSM/Sparkle 类服务端模组；必须复用 Figura 既有纯客户端 + 外部后端同步体系，这部分不作为当前阶段改造目标。 |

## 已基本一致的部分

### 1. 动画文件加载

Sparkle-Morpher 的模型装配会把 player 主模型动画、extra 动画、controller、projectile/vehicle 动画分包加载进 `ModelAssembly` / `PlayerModelBundle` / `ProjectileModelBundle` / `VehicleModelBundle`。

Figura 当前已做到：

- 从 `ysm.json -> files.player.animation` 收集动画路径。
- 支持 `main.animation.json`、`extra.animation.json`、`arm.animation.json`、`slashblade.animation.json`、`tac.animation.json` 等文件进入 `YsmAnimationPlayer`。
- 对旧版模型，已从 `main.json` 的 `ysm_extra_info.extra_animation_names` 生成轮盘动作。
- 动画同名冲突时会合并并尽量保留非空/内容更多的 clip。

差距较小，主要还在“动画控制器语义”而不是资源是否能读到。

### 2. 额外动作轮盘

Sparkle 的触发路径：

- `ExtraAnimationKey` 读取 `ModelProperties.extraAnimation`，按 0-7 快捷键触发。
- `UnifiedRouletteScreen` 支持根菜单、`#分类` 子菜单、`#return` 返回和自定义布局。
- 触发时发送 `C2SPlayAnimationPacket(index, category, key)`。

Figura 当前已做到：

- `YsmActionSchemaParser` 读取 `properties.extra_animation`。
- 支持 `extra_animation_classify` 数组/对象格式。
- 支持 `extra_animation_buttons` 生成配置项。
- `YsmModelRuntime.installDefaultActionWheel` 会把 extra 动作装进 action wheel。
- `YsmActionRuntime` 能触发、停止、查询动作，并已修正 active key 解析。

仍未完全一致的是 Sparkle 的自定义轮盘布局协议。Sparkle 的服务端动画同步协议不作为 Figura YSM native 的对齐目标；Figura 必须保持纯客户端模组 + Figura 外部后端同步的既有路线。

### 3. 默认隐藏与动作显示

Sparkle 语义：

- `pre_parallel` / `parallel` 可作为常驻基础控制层。
- 很多模型用 `pre_parallel0` 的 `scale: 0` 默认隐藏特效骨骼。
- extra 动作播放时用非零 scale 把这些骨骼拉出来，例如“坐水母”。

Figura 当前已对齐的点：

- 识别 `pre_parallel0..7`、`parallel0..7` 和带下划线路径形式。
- 零 scale 基础动画会进入 `baseHiddenBones`。
- 默认隐藏现在同步到 part default visibility。
- 非基础动作只要给出非零 scale，会重新显示该骨骼。
- 基础隐藏的零 scale 不再把后续动作乘成永久 0。

这部分已经按机制修复，不依赖骨骼名。

## 部分一致但需要继续对齐的部分

### 1. Animation Controller 体系

Sparkle 的 controller 注册非常细：

- player: `pre_parallel`、`vehicle`、`pre_main`、`main`、`post_main`、`pre_hold`、`hold_mainhand`、`hold_offhand`、`swing`、`use`、`passenger`、`cap`、`parallel`、`armor` 等。
- projectile: `pre_main`、`main`、`post_main`、`parallel`。
- vehicle: `pre_parallel`、`pre_main`、`main`、`move`、`origin`、`ride`、`post_main`、`parallel`。

Figura 当前：

- 有 `YsmControllerRuntime` 解析和运行 JSON controller。
- 有 native state machine 处理常见 player 状态。
- 有 controller function animation 机制。

差距：

- 没有完整复刻 Sparkle 的 controller slot registry。
- vehicle/projectile controller 还没有独立 runtime。
- transition length、begin/end transition、hold on last frame、play once 的行为仍比 Sparkle 简化。
- controller context 中的 `anim_time`、render layer、scale transition special 尚未完全对齐。

### 2. Timeline/Event 执行

Sparkle 的 `AnimationControllerInstance` 有：

- `InstructionKeyFrameExecutor`
- `SoundKeyFrameExecutor`
- loop 越界时 `executeRemainingEvents`
- beginning/ending transition 中的事件与上下文处理

Figura 当前：

- 支持 timeline 表达式事件，能执行 `v.roaming.xxx=...`。
- 支持基础 `sound`、`particle` 事件。
- 支持 0 秒 initial events 的分发。

差距：

- loop 边界、剩余事件补发、ending transition 时事件清算不完全。
- sound/particle 参数兼容范围小于 Sparkle。
- controller context 事件和 render layer 指令不完整。

### 3. Molang / Ctrl / YSM Binding

Sparkle 的 `CtrlBinding` 提供：

- `ctrl.idle/walk/run/sneak/jump/swim/fly/sleep/death` 等状态。
- `ctrl.playing_extra_animation`
- `ctrl.hold/use/swing/armor/ride/...`
- `ctrl.set_animation`
- `ctrl.set_beginning_transition_length`
- `ctrl.reset`
- 多个 mod compat 控制函数。

Sparkle 的 `YSMBinding` 提供大量查询：

- 装备、附魔、药水效果、光照、生物群系、天气、维度。
- 船/木筏/箱船桨状态和偏移。
- 武器类型、三叉戟、长枪、钉锤等动作状态。
- fishing、shoulder parrot、attack/swing、entity type 等。

Figura 当前：

- 已有 Molang binding，并补了部分 movement、ctrl、roaming、action/control 相关变量。
- 已能解析 controls 并把 `v.roaming.xxx` 映射到控制项。

差距：

- Sparkle 的 binding 覆盖面远大于 Figura。
- 很多模型 controller 条件会依赖 `ysm.*` 或 `ctrl.*` 的细变量，当前 Figura 可能默认为 0/false，导致动画不触发。
- `ctrl.reset`、`ctrl.set_animation`、transition length 这类函数式 controller 指令仍需系统对齐。

## 未完全对齐/高风险差异

### 1. Geckolib 骨骼动画队列与过渡

Sparkle 的核心是 `AnimationControllerInstance`：

- 每个 bone 有独立 queue。
- 支持 running、beginning transition、ending transition。
- 支持 transition snapshot。
- 支持 scale transition special。
- 支持 `PLAY_ONCE`、`LOOP`、`HOLD_ON_LAST_FRAME`。
- 事件执行和动画状态绑定在 controller context。

Figura 当前是简化模型：

- 每帧 reset part anim pose。
- 遍历 active animations，把 pos/rot 加权相加、scale 乘法叠加。
- fade 主要用于整体 weight，不是完整 controller queue。

这是目前最大机制差异。大多数简单模型可工作，但复杂动作、过渡、叠层 scale 和 controller-driven 动画会有偏差。

### 2. Projectile / Vehicle 子模型完整动画

Sparkle：

- `ModelAssembly` 持有 `projectileModels`、`vehicleModels`。
- `ProjectileModelBundle` / `VehicleModelBundle` 各自有 model、animations、animationControllers、texture。
- `ProjectileAnimationController` 和 `VehicleAnimationController` 有独立 controller pipeline。
- vehicle 有 `origin`、`move`、`ride` 等专用状态。

Figura 当前：

- 已读取 `files.projectiles` / `files.vehicles`。
- 已能把子实体模型和贴图载入 runtime。
- arrow/trident 已有静态渲染接入。

差距：

- projectile 子模型还未跑独立 animation/controller。
- vehicle 没有完整 entity renderer hook。
- vehicle origin/ride/move controller 未实现。
- 子实体 Molang context 未和真实 projectile/vehicle entity 完整绑定。

### 3. 网络同步与多人一致性

Sparkle 的额外动画触发会发包：

- 本地轮盘/快捷键触发 `C2SPlayAnimationPacket`。
- 服务端同步动作给其他客户端。
- 支持 entity id，非玩家实体也可触发。

这套机制不能直接迁移到 Figura：它隐含“目标 Minecraft 服务器必须安装并理解对应模组协议”的前提，而 Figura 的基本路线是纯客户端模组，通过 Figura 自己的外部后端完成多人可见能力。这个祖宗之法不能打破。

因此，当前阶段对网络同步的结论是：

- 不复刻 Sparkle-Morpher 的 `C2SPlayAnimationPacket` / 服务端转发协议。
- 不要求玩家所在 Minecraft 服务器安装任何 Figura YSM native 相关服务端模组。
- 如果后续需要同步 extra 动作状态，应接入 Figura 既有 action/ping/backend 同步链路，而不是新增依赖游戏服务器的协议。
- 这部分不是当前阶段考虑范围；当前阶段只保证本地 runtime、轮盘触发、controller/binding/子实体动画语义可落地。

## 逐项对齐建议

### 优先级 P0：继续修正基础动画/extra 动作语义

目标：保证所有“默认隐藏，动作显示”的模型稳定。

- 基础隐藏不应永久污染动作 scale。
- 动作停止后隐藏状态必须回到基础状态。
- active animation key 必须统一 resolve。
- 停止其他 extra 动作时必须清理旧动作残留。

当前已做大部分，仍建议用以下模型回归：

- `艺方原版娘化 发光鱿鱼娘`：坐水母、睡觉、03/04 pose。
- `08_sta`：车辆默认隐藏、离开坦克 timeline。
- `14_momo`：背包/小狐狸/持剑 controls。
- `15_kluonoa`：默认大小、行走动画和动作触发。

### 优先级 P1：补齐 controller slot registry

建议按 Sparkle 的 slot 分类逐步落地：

1. player: `pre_main/main/post_main/pre_parallel/parallel/cap`。
2. hold/use/swing/passenger/vehicle。
3. armor 和 mod compat 后置。
4. projectile: `pre_main/main/post_main/parallel`。
5. vehicle: `pre_parallel/pre_main/main/move/origin/ride/post_main/parallel`。

不要按具体模型名写分支，应按 slot/predicate 体系实现。

### 优先级 P1：补齐 Molang binding 覆盖面

建议建立对照清单：

- Sparkle `CtrlBinding` 中每个 var/function。
- Sparkle `YSMBinding` 中每个 var/function。
- Figura `MolangBindings` 中已有项。

然后分三类实现：

- 必须：动作触发和模型显隐依赖项。
- 常用：装备、主副手、移动、乘坐、船、攻击、钓鱼。
- 兼容：mod-specific 查询。

### 优先级 P2：实现 projectile/vehicle 独立动画 runtime

当前子模型只能静态显示，下一步应实现：

- `YsmSubEntity` 持有自己的 part map。
- 子实体动画 clip/controller 独立注册。
- 子实体渲染前按真实 entity state update。
- projectile/vehicle 各自绑定 Molang context。

### 优先级 P2：事件 executor 对齐

需要补齐：

- timeline loop 边界处理。
- play once 剩余事件补发。
- ending transition 中 sound/instruction reset。
- render layer / controller instruction 事件。

## 当前可判定为“完全或基本一致”的清单

- YSM 动画 JSON 的基础解析：`bones/position/rotation/scale/timeline/sound_effects/particle_effects`。
- `extra.animation.json` 进入 runtime。
- `extra_animation` / `extra_animation_classify` / `extra_animation_buttons` 进入轮盘和 controls。
- 旧版 `main.json` 的 `extra_animation_names` 动作生成。
- `pre_parallel0..7` / `parallel0..7` 基础 slot 名识别。
- `catmullrom` / `step` / easing 插值基础支持。
- 默认隐藏与 extra 非零 scale 显示的核心机制。
- arrow/trident 的 projectile 子模型资源加载和替换渲染入口。

## 当前明确未完全一致的清单

- Sparkle 的完整 controller slot registry。
- `AnimationControllerInstance` 的 bone queue 与 begin/end transition。
- `HOLD_ON_LAST_FRAME` 等 loop mode 的完整表现。
- controller context、render layer 指令、`ctrl.reset`、`ctrl.set_animation` 完整语义。
- Sparkle `YSMBinding` 大量实体/装备/武器/船/mod 查询变量。
- projectile 子模型独立动画/controller。
- vehicle 子模型 renderer hook、origin/move/ride controller。
- 基于 Figura 后端的 extra animation 多人同步适配，当前阶段暂不处理。
- sound/particle/resource event 的完整 executor 行为。

## 总体判断

Figura YSM 当前已经能覆盖一批普通 YSM 模型和部分复杂 extra 动作，但还不是 Sparkle-Morpher 的完整 YSM 引擎复刻。

最关键的剩余差距不是单个模型文件，而是三套通用机制：

1. controller slot/predicate registry；
2. Geckolib 风格的 bone animation queue 与 transition；
3. Sparkle 的 Molang binding 覆盖面和子实体独立动画 runtime。

后续修复应继续禁止模型名/骨骼名硬编码，按 slot、binding、controller、sub-entity 类型做通用实现。
